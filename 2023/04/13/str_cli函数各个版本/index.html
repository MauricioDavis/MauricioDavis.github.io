str_cli函数各个版本的区别及问题
2023-04-13 18:41:34 # programming #unix network programming

程序的功能是接受用户输入的数据并发送给服务器，接受回射数据进行输出


最初版本p100，交互式阻塞于fgets

unix网络编程中没有给出具体会出现问题的事件，只是提了可能会阻塞在fgets函数，这里进行了一些相关收集，总体感觉这段代码问题不大，除了后面提到的杀死僵死进程

搜索到的相关阻塞是

  1. stdout被重定向到其他文件且该文件以及满了则无法写入，会阻塞到第二个fgets
  2. stdout被重定向到不可写入的文件也会被阻塞

但基于书中程序的功能和上面不符所以感觉问题不大

第一次修订版本，阻塞与select函数p133

该程序使用了select函数进行阻塞，这样程序就不会阻塞在fgets函数，程序存在如下问题：

  1. 用户一次输出多条带\n的数据时只会回射输出一行(批量输入)，原因是程序使用的readline函数进行接收，所以接收到123\n就会停止接收，剩余的数据仍然在stdin中，而select函数是通过检查该套接字的接收缓冲区是否有数据来判断是否就绪，而下面的测试程序我们输入的py被从接收缓冲区复制到—->内核缓冲区再复制到—->用户缓冲区中，所以此时接收缓冲区是没有数据的，所以select函数不会认为准备就绪，直到第二次发送py，第二次发送是数据被复制到用户缓冲区中，但是第一次发送的数据依然存在所以我们第二次接收的是第一次的456\n

  1. 第二个问题是，该程序对EOF处理不当，该程序在用户输入完之后直接进行了返回到main函数并结束程序意味着所有套接字都被关闭了，但这里只能表明用户端输入完毕了并不代表服务器接收到了数据和自己接收到服务器的数据，所以该程序会导致自己还没有接受完毕数据就完全关闭了套接字，这个测试很简单，在服务端程序回射前sleep()且再客服端输入ctrl d表示输入结束，此时我们无法得到服务端的回射数据

这是服务端程序

程序并没有输出可无端的回射数据

第二次修订，select阻塞，使用shutdown进行半关闭

废弃了使用文本行为中心的代码。使用shutdown半关闭解决了第一次修订完全关闭套接字导致没有完全接收到服务器数据就关闭的情况

书本中的阻塞的str_cli函数讨论在我目前的学习进度中停止了，该断代码接收到ctrl d之后将使用shutdown函数关闭该套接字的发送也就是告知对端客服端发送已经中止，但是程序仍然可以接受服务端的数据，程序使用了一个标志判断select函数是否需要再次监测stdin

目前我这边认为该段程序目前是没有什么问题的，正确处理的EOF，使用read而不是readline只样就不会导致用户批量发送处理不准确的现象，不过这里不能下定论，现在我学习的还太浅
